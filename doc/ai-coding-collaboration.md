# AI协作规则

- 版本: v2.4
- 生效日期: 2026-05-13
- 最近变更: 完善门禁语义、质量检查、状态枚举、阶段字段、编码工具异常处理与规则变更历史。

## 0. 启动规则

- 本规则适用于用户显式要求开启需求跟踪、协作跟踪、Architect/Code协作文档或同等含义的受控协作流程。
- 用户未显式要求开启需求跟踪时，不强制创建协作文档。
- 用户未显式要求开启需求跟踪时，不强制将普通问答、代码解释、轻量修改建议或一次性分析结论落盘为协作文档。
- 一旦用户显式要求开启需求跟踪，后续该需求的 Architect 与 Code 输出必须写入同一份协作文档。
- 只允许使用 Architect 与 Code 两个角色。
- 禁止使用 Debug、Orchestrator、Ask 角色定义或职责。
- 输出目录固定为：`doc/`。
- 编码工具目录固定为：`encoding_tools/`。
- 需求跟踪开启后，任意 Architect 与 Code 阶段输出必须落盘为可复用协作文档。
- 需求跟踪开启后，单次需求只允许生成一份协作文档。
- 需求跟踪开启后，协作文档必须同时承载 Architect 章节与 Code 章节。
- 需求跟踪开启后，协作文档必须按角色分别建立独立章节。
- 需求跟踪开启后，协作文档必须使用固定章节名称与固定章节顺序。
- 需求跟踪开启后，每个需求必须定义唯一需求前缀。
- 需求前缀必须来自需求编号或需求名称。
- 需求前缀必须置于协作文档文件名前缀位置。
- 文件名格式必须为：`需求前缀-collaboration.md`。
- 命名示例：需求前缀为 `REQ-001` 时，协作文档必须命名为 `REQ-001-collaboration.md`。
- 禁止同一需求拆分多个正文文档。
- 禁止不同需求复用同一协作文档文件名。
- 需求跟踪开启后，禁止只在对话中给出最终结论。
- 需求跟踪开启后，无法创建输出目录时必须停止执行并记录阻塞原因。
- 需求跟踪开启后，无法写入文档时必须停止执行并记录阻塞原因。
- 需求跟踪开启后，无法使用 `encoding_tools/` 时必须停止执行并记录阻塞原因。
- 需求跟踪开启后，如需替代 `encoding_tools/` 执行源码修改，必须先由 Architect 在第1.4节 `Code任务执行包` 或第1.7节 `门禁裁判` 中明确允许。

## 1. Architect规则

- 需求跟踪开启后，Architect 必须创建并维护 `doc/需求前缀-collaboration.md`。
- Architect 必须负责初始化第1章 `Architect章节`。
- Architect 必须在第1章 `Architect章节` 中维护以下子章节：
  - 第1.1节 `需求定义`。
  - 第1.2节 `总体架构`。
  - 第1.3节 `单元设计`。
  - 第1.4节 `Code任务执行包`。
  - 第1.5节 `Architect需求跟踪矩阵`。
  - 第1.6节 `Architect关键接口跟踪矩阵`。
  - 第1.7节 `门禁裁判`。
- Architect 必须使用附录C.1模板创建协作文档。
- Architect 必须负责门禁裁判。
- Architect 必须输出门禁结论。
- Architect 门禁结论只允许为：通过、驳回、有条件通过。
- Architect 必须输出放行或阻塞结论。
- Architect 门禁结论为 `通过` 时，放行阻塞必须为 `放行`。
- Architect 门禁结论为 `驳回` 时，放行阻塞必须为 `阻塞`。
- Architect 门禁结论为 `有条件通过` 时，放行阻塞必须为 `放行`，并必须记录条件、责任方与关闭要求。
- Architect 不得在存在未处理阻塞项、未处理 Code任务反馈或不可接受质量风险时给出 `通过`。
- Architect 不得执行源代码增删改。
- Architect 不得删除 Code 已填写章节。
- Architect 不得在第2章 `Code章节` 中改写 Code 已提交的执行证据。

## 2. Code规则

- Code 必须只按同一协作文档中的第1.4节 `Code任务执行包` 执行。
- Code 不得自行扩展任务范围。
- Code 发现第1.4节 `Code任务执行包` 存在任务缺失、文件范围缺失、操作类型缺失、接口定义缺失、验收标准不可测试或规则冲突时，必须停止相关修改并在第2.6节 `Code任务反馈` 中记录反馈。
- Code任务反馈类型只允许填写：任务缺失、文件范围缺失、操作类型缺失、接口定义缺失、验收标准不可测试、规则冲突、执行阻塞。
- Code 提交任务反馈后，必须等待 Architect 更新第1章 `Architect章节` 或在第1.7节 `门禁裁判` 中给出裁决结论。
- Code 不得以任务反馈为依据自行修改第1章 `Architect章节`。
- Code 不得裁判门禁。
- Code 必须负责维护第2章 `Code章节`。
- Code 必须在第2章 `Code章节` 中维护以下子章节：
  - 第2.1节 `Code需求跟踪矩阵`。
  - 第2.2节 `Code关键接口跟踪矩阵`。
  - 第2.3节 `Code测试项跟踪矩阵`。
  - 第2.4节 `Code缺陷跟踪矩阵`。
  - 第2.5节 `Code执行证据`。
  - 第2.6节 `Code任务反馈`。
- Code 对源代码执行新增、删除、修改前，必须确认任务编号存在于第1.4节 `Code任务执行包`。
- Code 对源代码执行新增、删除、修改前，必须确认目标文件存在于第1.4节 `Code任务执行包`。
- Code 对源代码执行新增、删除、修改前，必须确认操作类型存在于第1.4节 `Code任务执行包`。
- Code 对源代码执行新增、删除、修改前，如任一确认项不存在，必须记录第2.6节 `Code任务反馈`，不得继续执行相关修改。
- Code 对源代码执行新增、删除、修改时，必须使用 `encoding_tools/README.md` 描述的接口。
- Code 对 C/C++ 源代码（`.c`、`.cc`、`.cpp`、`.cxx`、`.h`、`.hpp`）执行新增、删除、修改时，必须使用 `encoding_tools/encoding_safe_patch.py`。
- Code 对非 C/C++ 源代码执行新增、删除、修改时，可直接编辑，不强制使用 `encoding_tools/encoding_safe_patch.py`。
- Code 使用 `encoding_tools/` 失败时，必须记录失败命令、错误摘要、影响文件与阻塞影响，并在第2.6节 `Code任务反馈` 中提交反馈。
- Code 在 Architect 明确允许替代修改方式前，不得绕过 `encoding_tools/` 修改受控 C/C++ 源代码。
- Code 必须在第2.5节 `Code执行证据` 中记录修改接口、配置文件、执行报告、影响文件、测试命令、自测结果、未执行测试原因、遗留风险、回滚方案。
- Code 不得修改第1章 `Architect章节` 中由 Architect 已确认的裁判结论。

## 3. 文档规则

- 需求跟踪开启后的所有文档必须可复用。
- 需求跟踪开启后，单次需求只允许存在一份正文协作文档。
- 需求跟踪开启后，所有 Architect 与 Code 输出内容必须写入同一需求的协作文档固定章节。
- 需求跟踪开启后，协作文档必须按角色划分独立章节。
- 需求跟踪开启后，Architect 与 Code 必须分别在各自章节内维护输出内容。
- 需求跟踪开启后，所有输出文档必须使用附录模板。
- 需求跟踪开启后，所有输出文档必须遵守附录A公共模板规则。
- 需求跟踪开启后，所有文档必须使用稳定编号。
- 需求跟踪开启后，所有文档必须使用固定状态枚举：未开始、进行中、阻塞、待评审、待整改、已完成、已放行、已驳回。
- 需求跟踪开启后，所有文档文件名必须包含需求前缀。
- 需求前缀在同一需求内必须完全一致。
- 协作文档中的 Architect 章节与 Code 章节必须使用相同需求前缀。
- 需求前缀禁止使用空格。
- 需求前缀禁止使用路径分隔符。
- 所有矩阵必须保留需求编号。
- 所有接口矩阵必须保留接口编号。
- 所有测试项矩阵必须保留测试项编号。
- 所有缺陷矩阵必须保留缺陷编号。
- Architect 与 Code 的需求编号必须一致。
- Architect 与 Code 的接口编号必须一致。
- Code 不得新增 Architect 章节未声明的需求编号。
- Code 不得新增 Architect 章节未声明的接口编号。
- 任意章节更新必须同步到同一需求的协作文档。
- 任一协作文档缺少 `适用规则: AI协作规则` 时门禁必须驳回。
- 任一协作文档缺少 `后续工作传递声明` 时门禁必须驳回。
- 任一角色章节缺失时门禁必须驳回。
- 任一必需子章节缺失时门禁必须驳回。
- 任一章节标题与模板不一致时门禁必须驳回。

## 4. 门禁规则

- Architect 是唯一门禁裁判。
- Code 只能提交执行结果与证据。
- Architect 裁判时必须检查协作文档是否存在。
- Architect 裁判时必须检查第1章 `Architect章节` 与第2章 `Code章节` 是否存在。
- Architect 裁判时必须检查所有必需子章节是否存在。
- Architect 裁判时必须检查协作文档是否使用需求前缀。
- Architect 裁判时必须检查全文章节中的需求前缀是否一致。
- Architect 裁判时必须检查需求编号是否一致。
- Architect 裁判时必须检查接口编号是否一致。
- Architect 裁判时必须检查 Code 是否使用 `encoding_tools/`。
- Architect 裁判时必须检查 Code 证据是否包含修改接口、配置文件、执行报告、影响文件、测试命令、自测结果、未执行测试原因、遗留风险、回滚方案。
- Architect 裁判时必须检查第2.6节 `Code任务反馈` 是否已处理；存在未处理反馈时门禁必须驳回或有条件通过。
- Architect 裁判时必须检查验收标准是否可测试。
- Architect 裁判时必须检查每个需求是否至少关联一个 Code任务。
- Architect 裁判时必须检查每个 Code任务是否至少关联一个测试项或未测试原因。
- Architect 裁判时必须检查 Code 修改文件是否全部位于第1.4节 `Code任务执行包` 允许范围内。
- Architect 裁判时必须检查测试失败项是否已记录缺陷。
- Architect 裁判时必须检查未执行测试是否已说明原因。
- Architect 裁判时必须检查遗留风险是否已记录且可接受。
- Architect 裁判时必须检查所有模板必填字段是否存在。
- 门禁未通过时必须阻塞。
- 门禁阻塞后必须退回整改。
- 门禁未通过时禁止进入下一阶段。
- 有条件通过允许进入下一阶段，但条件关闭前不得关闭该需求。
- 有条件通过的条件必须在原协作文档中持续跟踪。
- 整改后必须在原协作文档中更新相关章节。
- 整改后必须重新执行 Architect 门禁裁判。

## 5. 执行顺序

1. Architect 定义需求前缀。
2. Architect 按附录C.1创建带需求前缀的协作文档。
3. Architect 完成第1章 `Architect章节` 下的第1.1节 `需求定义`。
4. Architect 完成第1章 `Architect章节` 下的第1.2节 `总体架构`。
5. Architect 完成第1章 `Architect章节` 下的第1.3节 `单元设计`。
6. Architect 完成第1章 `Architect章节` 下的第1.4节 `Code任务执行包`。
7. Architect 完成第1章 `Architect章节` 下的第1.5节 `Architect需求跟踪矩阵`。
8. Architect 完成第1章 `Architect章节` 下的第1.6节 `Architect关键接口跟踪矩阵`。
9. Architect 在第1章 `Architect章节` 下的第1.7节 `门禁裁判` 中给出阶段性裁判输入。
10. Code 按第1.4节 `Code任务执行包` 执行。
11. Code 使用 `encoding_tools/` 接口修改源代码。
12. Code 完成第2章 `Code章节` 下的第2.1节 `Code需求跟踪矩阵`。
13. Code 完成第2章 `Code章节` 下的第2.2节 `Code关键接口跟踪矩阵`。
14. Code 完成第2章 `Code章节` 下的第2.3节 `Code测试项跟踪矩阵`。
15. Code 完成第2章 `Code章节` 下的第2.4节 `Code缺陷跟踪矩阵`。
16. Code 完成第2章 `Code章节` 下的第2.5节 `Code执行证据`。
17. Code 如发现任务包缺口、范围冲突、接口缺失或验收阻塞，必须完成第2章 `Code章节` 下的第2.6节 `Code任务反馈`。
18. Architect 根据第2.6节 `Code任务反馈` 更新第1章相关章节，或在第1.7节 `门禁裁判` 中给出裁决结论。
19. Architect 在第1章 `Architect章节` 下的第1.7节 `门禁裁判` 中输出最终放行或阻塞结论。

## 6. 冲突规则

- 本规则优先于旧版协作规范。
- 当规则冲突时，必须停止执行并交由 Architect 裁判。
- 冲突必须记录冲突编号、冲突条款、最终采用条款、裁决人、裁决结论。
