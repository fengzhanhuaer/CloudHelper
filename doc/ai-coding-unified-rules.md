# AI协作统一规则 单一规范

## 1. 目标

建立一份可直接执行的统一规则，约束AI架构师、AI编码者、AI测试者在同一任务中的协作行为，避免重复强调要求。

## 2. 适用角色

- 架构师
- 编码者
- 测试者

## 3. 全局强制规则

1. 工作依据与规则传递声明必须显式写入
   - 任何基于本规则文档工作的AI（包括架构师、编码者、测试者）必须在其生成的当前文档中显式注明工作依据文档。
   - 必须显式注明所遵循的规则为本规则，并声明必须遵守本规则。
   - 必须显式声明后续接手者与协作者需遵守同一规则，不得降级或替换执行口径。
   - 未显式注明视为规则不符合，门禁判定不通过。

2. 文档必须显式落盘
   - 所有阶段产物必须写入`/doc/`对应角色目录。
   - 禁止仅在CLI对话中给结论而不输出文档。
   - 本规则仅适用于具备文件系统的执行环境，无法落盘视为门禁不通过。

3. 每份阶段文档必须包含统一最小内容
   - 工作依据与规则传递声明
   - 日期
   - 备注
   - 风险
   - 遗留事项
   - 进度状态
   - 完成情况
   - 检查表
   - 跟踪表状态
   - 结论记录

4. 需求跟踪表必须全程维护
   - 从需求进入开始建立。
   - 在架构、编码、测试、复盘阶段持续更新。
   - 必须记录需求编号、状态、责任角色、更新时间。

5. 文档边界
   - 该规范下的阶段文档仅记录治理信息与结论。
   - 不承载代码内容。
   - 若需引用实现细节，仅允许提交链接与差异摘要。

## 4. 角色规则

### 4.1 架构师规则

- 将用户需求细化为统一需求主文档。
- 完成关键选型并给出取舍依据。
- 首次承接需求时，必须显式询问并确认程序源代码编码格式要求。
- 输出总体设计、单元设计、接口定义，形成可实施设计基线。
- 将编码与测试任务拆解为执行单元包。
- 建立并维护需求跟踪表初版。
- 输出必须可直接驱动编码与测试执行。

架构师最低输出清单
- 统一需求主文档
- 关键选型决策记录
- 总体设计说明
- 单元设计说明
- 接口定义清单
- 执行单元包清单
- 编码与测试映射关系
- 需求跟踪表

### 4.2 编码者规则

- 严格按执行单元包实施。
- 必须按架构师明确指定的程序源代码编码格式实施，不得擅自变更。
- 每项实现必须映射到需求编号。
- 同步更新需求跟踪表中的编码状态。

编码者核查失败处理
- 若核查不通过，必须补齐缺失字段并回写跟踪表。
- 状态标记为阻塞，禁止流转到测试。

编码者最低输出清单
- 实现清单
- 编码阶段文档
- 自测结论
- 需求跟踪表编码状态更新

### 4.3 测试者规则

- 按测试单元包执行验证与回归。
- 每项测试必须映射到需求编号。
- 同步更新需求跟踪表中的测试状态与未通过处置状态。

测试者核查失败处理
- 若核查不通过，必须记录失败项与处置计划并回写跟踪表。
- 状态标记为待修复，禁止进入复盘。

测试者最低输出清单
- 测试记录
- 测试阶段文档
- 回归结论
- 未通过项处置记录
- 需求跟踪表测试状态更新

## 5. 统一流程规则 S1到S5

- S1 需求澄清
  - 最小启动输入允许仅有用户需求与关键选型偏好。
  - 输出需求文档与需求跟踪表初版。

- S2 架构设计
  - 输出关键选型结论、总体设计、单元设计、接口定义、执行单元包、映射关系。
  - 更新需求跟踪表设计映射。

- S3 编码与自测
  - 输出实现清单、编码阶段文档、自测结果。
  - 更新需求跟踪表编码状态。

- S4 测试与修复验证
  - 输出测试记录、测试阶段文档、回归结论。
  - 更新需求跟踪表测试状态。

- S5 复盘与回灌
  - 核验跨角色输出闭环。
  - 确认需求跟踪表状态闭环。

## 6. 门禁规则

### 6.1 门禁裁判与判定口径

- 门禁裁判角色：架构师。
- 架构师负责G1到G5的通过与不通过判定，并记录判定理由。
- 未获得架构师明确通过，不得流转到下一阶段。
- 门禁结果仅允许两种状态：通过、不通过。

### 6.2 检查表统一结构

- 检查项编号
- 检查项描述
- 检查结果
- 证据链接
- 责任角色
- 核查日期
- 处置动作
- 备注

### 6.3 状态枚举

- 进度状态：未开始、进行中、已完成、阻塞
- 跟踪表状态：待分析、待设计、待实现、实现中、待测试、测试中、待修复、已闭环

- G1 需求门
  - 需求目标、范围、验收标准完整。
  - 跟踪表初版已建立。

- G2 架构门
  - 关键选型与取舍依据完整。
  - 总体设计、单元设计、接口定义完整。
  - 执行单元包与映射关系完整。
  - 已记录架构师首次显式询问编码格式与确认结果。
  - 架构阶段文档中的工作依据与规则传递声明完整。

- G3 编码核查门
  - 实现清单完整。
  - 编码阶段文档统一最小内容完整。
  - 编码实现与架构师指定的编码格式一致。
  - 编码阶段文档中的工作依据与规则传递声明完整。
  - 跟踪表编码状态与实现清单一致。

- G4 测试核查门
  - 测试记录与回归结论完整。
  - 测试阶段文档统一最小内容完整。
  - 测试阶段文档中的工作依据与规则传递声明完整。
  - 跟踪表测试状态与测试记录一致。

- G5 复盘门
  - 改进项已登记。
  - 跟踪表状态闭环。

## 7. 异常升级规则

- P0 核心阻断
  - 立即升级到架构师，触发跨角色应急协同。

- P1 关键退化
  - 记录处置计划并重新过门禁。

- P2 一般缺陷
  - 纳入迭代并在复盘闭环。

## 8. 执行口径

当用户提出新需求时，三类AI必须先对照本规则执行，不得跳过文档落盘、跟踪表更新与门禁核查，且所有阶段流转以架构师门禁判定为准。任一阶段文档未显式写明工作依据与规则传递声明时，视为门禁不通过并退回上一门禁重审。

## 9. 程序源代码编码格式章节

1. 首次询问责任
   - 架构师在首次承接需求时，必须显式询问用户或需求提出方的程序源代码编码格式要求。
   - 询问至少覆盖：命名风格、注释规范、格式化工具、静态检查规则、目录与文件组织约定。

2. 编码格式基线建立
   - 架构师必须将确认后的编码格式写入架构阶段文档，形成“编码格式基线”。
   - 未确认前不得默认跳过；若用户未给出明确格式，需在文档中显式记录采用的默认格式及依据。

3. 编码执行约束
   - 编码者必须严格按照架构师指定的“编码格式基线”编写实现。
   - 编码者发现格式冲突或缺失时，必须回传架构师补充确认后再继续实现。

4. 变更控制
   - 编码格式一经基线确认，任何调整必须由架构师显式批准并更新文档与跟踪表。
   - 未经批准的格式变更视为不符合，编码核查门不通过。

5. 门禁关联
   - G2必须核验“首次显式询问与编码格式基线记录”是否完整。
   - G3必须核验“编码实现与编码格式基线一致性”是否通过。
