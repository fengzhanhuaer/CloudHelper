# 网络助手技术可行性分析（基于当前代码）

## 1. 总体结论

- 总体可行：`可行（分阶段落地）`
- 当前基础能力已具备：本地代理、白名单直连、WebSocket 隧道、会话鉴权、Windows TUN 基础。
- 主要短板：规则引擎未上线、代理组/级联未实现、链路管理与 NAT 端口映射模型未建立、Linux TUN 未实现、多承载协议未抽象。

## 2. 现状基线（代码事实）

- 管理端已实现 `direct/global/tun` 模式切换与本地 SOCKS/HTTP 代理。
- `direct_whitelist` 已落地到本地文件，并支持 `CIDR/IP/host` 解析。
- 隧道当前走 `WebSocket + yamux`，服务端隧道路由当前固定为 `cloudserver`。
- 隧道与管理接口要求 HTTPS 且校验会话 token。
- 前端已明确“规则模式 V2 开放”，说明规则能力尚未交付。
- TUN 当前仅支持 Windows amd64；非 Windows 下系统代理自动配置也未实现。

## 3. 按需求逐条可行性

1. 按规则代理本机出站流量，直连或探针出口  
可行性：`中高`  
现状：已有“白名单直连 + 其余走隧道”雏形。  
缺口：缺完整规则引擎（优先级、动作、冲突处理、持久化）。

2. 探针出口可为直接出口或多级探针级联出口  
可行性：`中`  
现状：已有探针会话与隧道复用能力。  
缺口：缺“级联路由编排”和“多跳转发执行面”。

3. 直连通过 `direct_whitelist` 定义  
可行性：`已实现（可增强）`  
现状：已读取并解析 `direct_whitelist.txt`，命中后直连。  
缺口：缺 UI 管理、热更新反馈与更细粒度校验。

4. 规则支持 `CIDR/IP/DOMAIN 后缀`，并支持按代理组配置  
可行性：`中高`  
现状：已支持 `CIDR/IP/host`（host 目前是精确匹配）。  
缺口：`DOMAIN 后缀`、代理组模型、规则与代理组绑定尚无实现。

5. 代理组是一组探针出口，由用户在界面选择其一  
可行性：`高`  
现状：已有节点选择 UI 与状态同步机制。  
缺口：代理组实体、组内选择策略、前端配置入口。

6. 本地与远方探针使用私有协议，承载链路可选 `TCP/HTTPS/HTTP3/WebSocket` 其一  
可行性：`中`  
现状：已有私有帧协议语义（open/relay）并跑在 WebSocket+yamux。  
缺口：缺“承载层抽象接口”，目前仅 WS 路径；HTTP3/纯TCP/纯HTTPS 均需新增实现。

7. 链路管理：新建/删除/修改探针出口、设置链路协议、级联各跳协议可不同  
可行性：`中`  
现状：探针节点管理能力已有基础。  
缺口：未形成“链路实体”与“逐跳协议配置”模型；缺对应 UI 与运行时编排。

8. HTTPS 提供 codex 鉴权接口，成功后进入私有协议，失败延时返回失败响应  
可行性：`中高`  
现状：已具备 HTTPS + session token 鉴权链路。  
缺口：缺独立 codex 鉴权入口语义、失败延时策略、统一错误码约定。

9. Windows 采用 TUN 网卡，Linux 使用 TUN  
可行性：`中`  
现状：Windows TUN 具备安装与启用基础。  
缺口：Linux TUN 数据面、路由与 DNS 接管、权限与发行版差异适配尚未实现。

10. 兜底规则界面可选直连、代理组、拒绝  
可行性：`高`  
现状：已有模式/节点 UI 交互框架。  
缺口：规则兜底动作模型与“拒绝”执行逻辑、界面配置项。

11. 直连链路或级联链路在拓扑设置完成后自动保活并供业务复用  
可行性：`中高`  
现状：已有 `WebSocket + yamux` 长连接与多路复用基础。  
缺口：缺按 `path_id` 组织的会话池、预热流程和与拓扑快照联动的保活策略。

12. 探针服务接口通过链路管理设置，并支持 `NAT` 场景下服务端口与公网端口手动不一致  
可行性：`中`  
现状：当前只有固定隧道路由与节点 ID 选择，尚无“服务接口/公网映射”配置实体。  
缺口：缺出口端点模型、`service_endpoint/public_endpoint` 字段、NAT 手工映射校验，以及运行态对最终拨号地址的观测。

## 4. 关键风险

- 架构风险：隧道服务端路由当前固定为 `cloudserver`，多出口与级联需要先解耦路由入口。
- 建模风险：若不显式区分探针服务端口与 NAT 公网端口，链路管理很容易错误假设“内外端口一致”，导致连接失败且难以排查。
- 协议风险：多承载并存会放大心跳、重连、流控、观测的一致性成本。
- 系统风险：Linux TUN 涉及 root 权限与发行版差异，属于高运维复杂度模块。
- 安全风险：鉴权失败“延时返回”需做统一策略，避免放大 DoS 成本。

## 5. 建议落地顺序

1. 先做规则 MVP：`direct_whitelist` 增强、`DOMAIN 后缀`、兜底动作、代理组选择（不做级联）。
2. 再做出口与链路管理：探针出口/代理组/链路配置模型、探针服务接口与 NAT 公网端口映射模型和 UI。
3. 再做级联：先单链路级联，再扩展多跳与逐跳异构协议。
4. 最后做多承载与 Linux TUN：作为独立里程碑推进。

## 6. 当前判断

- 若按“先规则与单跳、后级联与多承载”的路线，项目技术风险可控。
- 若一次性并行上“级联 + 多承载 + Linux TUN”，交付风险明显升高，不建议。
